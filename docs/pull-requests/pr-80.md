# [PR 80](https://github.com/vig-os/commit-action/pull/80) build(devkit): adopt devkit 1.2.1 to fix release automation

## What & why

Adopt vigOS devkit `1.2.1` to unblock the **0.3.0** release, which failed in
the automated release workflow (**Fixes #79**).

**Root cause:** the scaffolded `release-core.yml` applied the draft + approval
gate to *every* release kind, so a `release_kind=candidate` dispatch aborted
against its still-draft PR (`ERROR: PR #N is still in draft`). devkit `1.2.1`
guards that gate behind `release_kind=final` — candidates gate on CI only, the
`main`-protecting approval is re-enforced at promote (vig-os/devkit#1095).

## Changes

- **`release-core.yml`** — regenerated from the 1.2.1 scaffold; the draft +
  approval gate is now `release_kind=final`-only (the #79 fix).
- **`flake.nix` / `flake.lock` / `.vig-os`** — `vigos` input and
  `DEVKIT_VERSION` bumped to `1.2.1` (kept in sync per devkit#1093).
- **Reconciled the #76 manual workarounds now fixed upstream (devkit#1092):**
  - Removed the bespoke `dist/.gitignore`; the managed `.gitignore` now ignores
    the `dist/src/` `tsc`/`ncc` declaration byproducts while keeping
    `dist/index.js` + `dist/package.json` tracked.
  - Moved the flake-generated `.pre-commit-config.yaml` symlink ignore to the
    new consumer-owned **`.gitignore.project`** (the #1092 auto-seed only fires
    when the store symlink resolves at scaffold time, which a direnv-mode
    container upgrade can't do).
- **Restored the #62 tag scheme in `.vig-os`** — the 1.2.1 scaffold reset
  `DEVKIT_TAG_PREFIX` and `DEVKIT_FLOATING_TAGS` to defaults; restored
  `v` / `major,minor` so 0.3.0 still ships `vX.Y.Z` tags and moves `v0` / `v0.X`.
- **CHANGELOG** — updated the `[0.3.0]` devkit bullet to reflect shipping on
  1.2.1 and the #79 fix.

## Verification

- `just precommit` (full `prek` suite) green locally, including
  `no-commit-to-branch`.
- Bundle rebuilt with the pinned toolchain (`npm ci && npm run bundle`);
  `dist/index.js` did not drift — the only `dist/` change is the intentional
  `dist/.gitignore` removal (`dist-check` parity).

## Note

`gh issue develop` generated a `build/79-…` branch name, which the enforced
branch-naming hook rejects (`build/` isn't an allowed prefix; dots aren't valid
in the slug). This PR uses the compliant `chore/79-adopt-devkit-1-2-1`; the
empty `build/79-adopt-devkit-1.2.1` branch it left on the remote can be deleted.
