# [PR 34](https://github.com/vig-os/commit-action/pull/34) refactor(commit): apply retry per REST call site in createTree

## Summary

Retries are now applied at **individual Octokit REST call sites** inside `createTree` / `createTreeChained` / `createBlob`, replacing the coarse `withRetry` wrap around the *entire* `createTree` in `commitViaAPI`. A transient mid-batch failure now retries only the failing call instead of re-running the whole multi-call operation and re-uploading already-created blobs/trees.

Closes #24.

## How
- New exported `RetryOptions { config: RetryConfig; logger? }` + private `callWithRetry(fn, retry?)` helper — calls `fn()` once when `retry` is undefined, else `withRetry(fn, config, logger)`.
- `retry?: RetryOptions` threaded (optional, defaults to **no retries** → backward-compatible signatures) through `createBlob`, `createTreeChained`, `createTree`.
- `commitViaAPI` drops the `withRetry(() => createTree(...))` wrap and passes `{ config: retryConfig, logger: log }` into `createTree`. The other single-call coarse wraps (getCommit/getBranchInfo/createCommit/updateBranch) are unchanged (already one call each).

## Tests (both fail against the old coarse wrap)
1. **Chunked tree retry**: 101 files → 2 chunks; chunk-2 fails 503 once then succeeds → `createTree` called exactly 3× (not 4), retried call still carries `base_tree = tree-chunk-1` (chunk-1 not re-uploaded).
2. **Blob retry**: blob-2 fails 503 once → `createBlob` called 3× (blob-1 once), single `createTree` carries both resolved SHAs.
