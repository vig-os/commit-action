## [Comment #2](https://github.com/vig-os/commit-action/pull/50#issuecomment-4967347932) by [@c-vigo](https://github.com/c-vigo)

_Posted on July 14, 2026 at 09:17 AM_

Closing in favour of **#58** (migrate action to ESM).

Two independent breakages, neither fixable as a dependency bump:

**1. ESM-only packaging.** `@actions/github` v9 is `"type": "module"` with no `"require"` export condition, so `ncc` cannot bundle it:

```
Error: Module not found: Error: Package path . is not exported from package @actions/github
```

**2. `moduleResolution: "node"` ignores `exports` maps.** `@actions/github`'s own `.d.ts` imports `@octokit/core/types`, which `@octokit/core@7` exposes *only* via its `exports` map. Classic resolution silently fails to find it, which collapses `octokit.rest.*` to `any` — and that is what surfaces as the three `TS2339` errors when `any` leaks into the generic `withRetry<T>` wrapper:

```
src/commit.ts:154:11 - error TS2339: Property 'data' does not exist on type 'unknown'.
src/commit.ts:208:13 - error TS2339: Property 'data' does not exist on type 'unknown'.
src/commit.ts:451:13 - error TS2339: Property 'data' does not exist on type 'unknown'.
```

**Worth flagging from (2):** the Octokit REST surface in `src/commit.ts` is currently *unchecked*. Call sites that don't go through the retry wrapper (`createCommit`, `getRef`, `getCommit`) don't error — they're silently `any`. Fixing `moduleResolution` restores real types and may surface latent errors; captured in #58, and it's the point of the exercise rather than a regression. Recompiling with `--moduleResolution bundler --lib ES2022` gives zero type errors project-wide.

Ruled out the `overrides` pins directly: loosening `@actions/http-client` to `^4.0.0` and `undici` to `6.27.0`, reinstalling, and re-running gave byte-identical failures.
