# [PR 74](https://github.com/vig-os/commit-action/pull/74) build(esm): migrate action and test suite to ESM

Refs #58. Closes #49, #50, #51. Unblocks #54.

The `@actions/*` toolkit is ESM-only at its next major — its `exports` map has no `"require"` condition at all — so `@actions/core` v3, `@actions/github` v9 and `@actions/http-client` v4 cannot land individually. They move together here.

## Changes

- **`package.json`** — `"type": "module"`. Dropped the `@actions/http-client` and `undici` `overrides`: the `http-client` pin (`3.0.2`) was what actually blocked the v4 major, and the `undici` pin is now redundant, since `http-client` v4 depends on `undici: ^6.23.0` which resolves to the same `6.27.0` the pin was added to force. `npm audit --omit=dev` reports 0 vulnerabilities.
- **`tsconfig.json`** — `module`/`moduleResolution` → `nodenext`; relative imports carry explicit `.js` extensions.
- **Bundling** — ncc handles ESM: it emits an ESM `dist/index.js` **plus a `dist/package.json`** declaring the bundle's module type. That second file is real shipped output, so it is now tracked and re-admitted in `.gitignore` (which #70 had just narrowed to `dist/index.js` alone — it would otherwise have been silently dropped).
- **Jest** — ESM mode (`--experimental-vm-modules`). Mocks move from `jest.mock` + a `__mocks__` manual mock to `jest.unstable_mockModule` + dynamic imports; `src/__tests__/setup.ts` is deleted and each suite registers its own mocks. **All 84 tests preserved** — `describe`/`it` titles and `expect()` counts are byte-identical to `HEAD`, no skips.

## Two latent problems this surfaced — both invisible to the test suite

**1. The bundled action was a silent no-op.** The CJS `require.main === module` entry guard does not survive bundling to ESM: ncc rewrites it to `__nccwpck_require__.c[__nccwpck_require__.s] === module`, which evaluates false. Executing that bundle:

```console
$ node dist/index.js          # GITHUB_TOKEN, TARGET_BRANCH, ... all set
$ echo $?
0
```

No output. `main()` never ran — the action would have **exited 0 having committed nothing**. Replaced with an ESM guard comparing `import.meta.url` against `pathToFileURL(process.argv[1])`. Same invocation now:

```console
$ node dist/index.js
Using TARGET_BRANCH: dev
No files to commit
```

Nothing in CI would have caught this: unit tests import `main()` directly, and the repo dogfoods `commit-action` at a pinned released SHA, so no workflow ever executes the locally-built bundle.

**2. The signed-commit REST flow was never type-checked.** Classic `moduleResolution: node` ignores `exports` maps, so `@octokit/core/types` never resolved and every `octokit.rest.*` call in `src/commit.ts` silently degraded to `any`. Probe on the old config — `r.data.sha` typed as `any`; under `nodenext` it types as `string`. The flow now type-checks clean against real Octokit types (no latent errors surfaced — `commit.ts` was already correct, just unverified).
