# [PR 86](https://github.com/vig-os/commit-action/pull/86) ci: add temporary Dist Check bootstrap stub

Refs #85 (removal tracking).

## Why

The `Main protection` and `Release protection` rulesets require a passing **`Dist Check`** status (GitHub Actions, integration `15368`). The real check (`dist-check.yml`, #59) ships with **0.3.0** and is **not yet on `main`**, so any PR whose head predates it — e.g. #82, and other branches cut from the current `main` — can never report the context and is permanently blocked.

## What

A temporary bootstrap workflow, `dist-check-stub.yml`, with a single job named exactly **`Dist Check`** (the contractual context name, #60) that always passes, triggered on `pull_request` to `main` and `release/**`. It unblocks that class of PRs.

## Deliberate design

- **Distinct filename** (not `dist-check.yml`): sharing the real path would cause an add/add merge conflict when `promote-release` merges `release/0.3.0 -> main`, breaking the automated promotion.
- **Self-satisfying:** because it triggers on `pull_request`, this PR's own head reports `Dist Check`, so — together with `CI Summary` from `ci.yml` and your review fix — this PR merges through the ruleset **without** a force/bypass.

## Lifecycle (important)

This is a scaffold that verifies nothing. **It must be deleted the moment 0.3.0 is promoted to `main`** — once the real `dist-check.yml` is there, both report `Dist Check` and the always-green stub would mask a real failure. Tracked in #85.

Local checks: `actionlint` clean; `just precommit` (prek `--all-files`) green.
