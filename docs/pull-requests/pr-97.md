# [PR 97](https://github.com/vig-os/commit-action/pull/97) ci: poll scratch-branch head in smoke verify steps to absorb replication lag

## Summary

Fixes the read-after-write race in both smoke workflows' verify steps (#96).

The `e2e-smoke` and `published-tag-smoke` "branch head points at the reported commit" checks read the scratch-branch head via `gh api .../commits/${SCRATCH_BRANCH}` **immediately** after the action's `updateRef` returns. GitHub's replication lag can serve the stale pre-commit head, so the check fails even though the action committed correctly — exactly what happened in [run 29433505922](https://github.com/vig-os/commit-action/actions/runs/29433505922), which an unchanged re-dispatch then passed.

## Change

Both verify steps now poll the branch head for the reported `commit-sha` with a bounded retry (**up to 5 attempts, 3s apart**) before declaring a mismatch. The hard failure after the retry budget is preserved, so a genuine no-op (the #58 class of bug this assertion exists to catch) still fails the job.

Steps 1 (empty `commit-sha`), 3 (signature), and 4 (content round-trip) are unchanged.

## Testing

- `prek run --all-files` green (yamllint, shellcheck, etc.).
- The retry path exercises on the next PR/tag smoke run.
