# [PR 76](https://github.com/vig-os/commit-action/pull/76) build(devkit): adopt devkit 1.2.0 and declare the v tag scheme

## What & why

Adopt vigOS devkit `1.2.0` and declare the per-repo release tag scheme that issue #62 was blocked on. 1.2.0 ships both blockers (`DEVKIT_TAG_PREFIX` vig-os/devkit#1044 and `DEVKIT_FLOATING_TAGS` vig-os/devkit#1045), so `.vig-os` now declares:

```ini
DEVKIT_TAG_PREFIX=v          # restore v-prefixed pipeline tags (vX.Y.Z)
DEVKIT_FLOATING_TAGS=major,minor  # auto-move v0 / v0.X at promote
```

Both floating tags were previously moved by hand.

## Deployment was broken out of the box — three regressions found & fixed

Re-scaffolding onto 1.2.0 surfaced three fresh breakages (not in the release notes), each caught by running the real hook suite:

1. **`typos` scanned the vendored bundle.** The language-aware managed `.gitignore` (devkit #1024) dropped this repo's `dist/` handling, so `typos` ran over `dist/index.js` and failed on typos inside undici/octokit. → Re-ignored `dist/src/` byproducts via a new, upgrade-proof nested `dist/.gitignore`.
2. **`check-json` rejected `.vscode/settings.json`.** The scaffold added a JSONC provenance banner (#1053) but the compensating `check-json` exclude ships via the `vigos` flake input, still pinned to 1.1.0 — a silent version skew. → Bumped `vigos.url` to `?ref=1.2.0` and `nix flake update vigos`.
3. **The flake-generated `.pre-commit-config.yaml` (`/nix/store` symlink) was no longer gitignored** and would have been committed broken. → Re-ignored in the managed root `.gitignore` (the only committed mechanism for a repo-root path).

Root causes filed upstream: vig-os/devkit#1092 (managed `.gitignore` drops consumer-required ignores) and vig-os/devkit#1093 (version skew ships the banner without its exclude).
