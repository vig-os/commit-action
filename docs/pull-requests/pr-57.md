# [PR 57](https://github.com/vig-os/commit-action/pull/57) chore(deps): batch six Renovate dev-dependency updates

## Batch six Renovate dev-dependency PRs into one

Supersedes **#45, #46, #47, #48, #55, #56** — all green individually, all
mutually conflicting.

### Why batch

Renovate opens one PR per npm package, and every one of them touches both
`package-lock.json` and `CHANGELOG.md`. Once #41 merged, all six went
`CONFLICTING/DIRTY` against `dev` *and* against each other. Landing them serially
means six rounds of rebase → regenerate lockfile → re-run CI → merge, where each
merge re-dirties the remaining five.

Batching regenerates the lockfile **once** off `dev`, rather than hand-resolving
`package-lock.json` conflicts six times. One CI run, one merge.

Built by applying the bumps to a fresh branch off `dev` (not by merging the six
Renovate branches, which would have meant six lockfile conflict resolutions).

### Security side effect

`ts-jest` `29.4.11` no longer depends on `handlebars` — **this clears the only
`critical` advisory.** Known dev-dependency vulnerabilities drop **10 → 5**:

```
before:  critical=1  high=6  moderate=2  low=1     (10)
after:   critical=0  high=2  moderate=2  low=1     ( 5)
```

Remaining: `@babel/core`, `brace-expansion`, `flatted`, `js-yaml`, `picomatch` —
all reachable only through `eslint` 8, so they clear with the ESLint 10 / flat-config
migration, not here.

### Not in this PR

- **`dist/` is deliberately not rebuilt** — the release cycle owns it. Note the
  `@vercel/ncc` bump changes the bundler, so the next `dist/` rebuild will produce
  a different bundle; that is expected and belongs to that cycle.
- **#53 (eslint v10)** stays out — it needs the `.eslintrc.json` → flat-config
  migration, since ESLint 10 removed the legacy config path. This PR's
  `typescript-eslint` v8 bump is a *prerequisite* for it (v6 hard-crashes under
  ESLint 10 with `TypeError: scopeManager.addGlobals is not a function`).
- **#49 / #50 / #51 (`@actions/*` majors)** stay out — all three are ESM-only and
  need a coordinated ESM migration; `ncc` cannot bundle them into the current CJS
  build. Tracked separately.
