# [PR 35](https://github.com/vig-os/commit-action/pull/35) feat(commit): make createTree chunking payload-size aware

## Summary

`createTreeChained` chunked by **entry count only** (`TREE_ENTRY_CHUNK_SIZE = 100`), so a chunk of many large inline text files could approach GitHub's ~100 MB request-body limit. This adds payload-size awareness and a per-file blob-fallback.

Closes #22.

## Changes
- **Byte-aware chunking** — new exported `TREE_ENTRY_BYTE_LIMIT = 6 MiB`. `createTreeChained` rewritten from a fixed-stride slice loop to an accumulating loop with a `flush()` helper: a chunk ends when adding the next entry would breach **either** the count cap **or** the byte budget (whichever first). Empty-chunk guard ensures a single oversized entry still makes progress. Chaining semantics (each chunk's tree SHA → next `base_tree`) unchanged.
- **Per-file threshold** — new exported `INLINE_CONTENT_SIZE_LIMIT = 1 MiB`. Text files larger than this go through `createBlob` (base64) instead of inline `content`.
- `estimateEntrySize` — safe over-approximation (64 B overhead + path + content/sha bytes).

## Tests
1. 4× 2.5 MiB text files (under the count cap) split at 2 entries/chunk purely on the byte budget → 2 `createTree` calls, correct `base_tree` chaining, `createBlob` never called.
2. One text file at `INLINE_CONTENT_SIZE_LIMIT + 1` → routed through `createBlob`, tree entry carries `sha` not `content`.
